> 每条场景描述：前置条件、操作序列、期望消息序列、判定标准。
> 配套文件：《自动化测试计划.md》说明如何用脚本客户端自动化执行这些场景。

## 详细场景

### 场景 1：ping / pong 心跳

**覆盖点**：可选消息兼容（公共接口"选择可选消息的一方应主动兼容没有可选内容的一方"）

**前置条件**：客户端已连接服务器（WebSocket 握手完成）。

**操作序列**：
1. 客户端发送 `{"messageType":"ping","timestamp":1712345678901}`
2. 服务器应答 pong

**期望消息**：
- 服务器回复：`{"messageType":"pong","timestamp":1712345678901}`（timestamp 原样返回）

**判定标准**：客户端收到 pong，timestamp 与发送值一致。

---

### 场景 2：Login 成功

**覆盖点**：认证主干，loginResult 消息格式

**前置条件**：
- 服务器启动（默认 8887 端口）
- 用户 `u1` 已注册（密码 `123456`），或服务器开启 `autoRegisterOnLogin=true`（默认）

**操作序列**：
1. 客户端连接 WebSocket
2. 发送 `{"messageType":"Login","userId":"u1","password":"123456"}`

**期望消息**：
```json
{"messageType":"loginResult","success":true,"message":"ok","userId":"u1"}
```

**判定标准**：
- `success` = true
- `userId` = "u1"
- 客户端状态变为 AUTHED

---

### 场景 3：register 成功

**覆盖点**：注册+认证，用户可随后登录

**前置条件**：服务器启动，用户 `u2` 不存在。

**操作序列**：
1. 客户端连接 WebSocket
2. 发送 `{"messageType":"register","userId":"u2","password":"123456","nickname":"测试用户"}`

**期望消息**：
```json
{"messageType":"loginResult","success":true,"message":"ok","userId":"u2"}
```

**判定标准**：
- `success` = true
- 同一客户端可随后用 `u2` + `123456` 登录成功

---

### 场景 4：重复登录拒绝

**覆盖点**：错误码 1002（重复登录）

**前置条件**：用户 `u1` 已在线（场景 2 完成）。

**操作序列**：
1. 新建第二客户端连接 B
2. B 发送 `{"messageType":"Login","userId":"u1","password":"123456"}`

**期望消息**：
```json
{"messageType":"error","code":1002,"message":"duplicate login"}
```

**判定标准**：收到 error code=1002，B 未被认证。

---

### 场景 5：startMatch → matchSuccess

**覆盖点**：匹配主干

**前置条件**：
- 客户端 A 已登录（userId="u1"）
- 客户端 B 已登录（userId="u2"）
- 双方均未在匹配中

**操作序列**：
1. A 发送 `{"messageType":"startMatch"}`
2. （A 进入等待队列）
3. B 发送 `{"messageType":"startMatch"}`

**期望消息**：
- B 收到：
  ```json
  {"messageType":"matchSuccess","roomId":"room_1","opponentId":"u1","opponentNickname":"u1"}
  ```
- A 收到：
  ```json
  {"messageType":"matchSuccess","roomId":"room_1","opponentId":"u2","opponentNickname":"u2"}
  ```

**判定标准**：
- 双发 matchSuccess，roomId 相同
- opponentId 互为对方

---

### 场景 6：双 Ready → gameStart

**覆盖点**：准备就绪 + 开局

**前置条件**：场景 5 完成，A 与 B 在同一房间。

**操作序列**：
1. A 发送 `{"messageType":"Ready"}`
2. B 收到 roomInfo（opponentReady=true）
3. B 发送 `{"messageType":"Ready"}`
4. A 收到 roomInfo（opponentReady=true）
5. 双方收到 gameStart

**期望消息**：
- 第二步 B 收到：`{"messageType":"roomInfo","opponentReady":true}`
- 第五步双方各收到 gameStart：
  - A（红方）：`yourColor="red"`, `firstHand=true`
  - B（黑方）：`yourColor="black"`, `firstHand=false`

**判定标准**：
- 双发 roomInfo 后双发 gameStart
- yourColor 互异，firstHand 正确（红方=true）

---

### 场景 7：gameStart 字段完整性校验

**覆盖点**：定制化字段正确性，initialBoard 格式

**前置条件**：场景 6 完成。

**操作序列**：收到 gameStart 后逐字段校验。

**期望字段**：
| 字段 | 类型 | 说明 |
|---|---|---|
| messageType | string | "gameStart" |
| redPlayerId | string | 红方 userId |
| blackPlayerId | string | 黑方 userId |
| yourColor | string | "red" 或 "black" |
| firstHand | bool | 己方是否为先手 |
| initialBoard | array（可选） | 32 个棋子格 |

**initialBoard 每格**：
| 字段 | 类型 | 说明 |
|---|---|---|
| x | string | 列 a–i |
| y | int | 行 0–9 |
| piece | string | 将帅="king"、visible=true；暗子=该格虚拟类型、visible=false |
| visible | bool | 明/暗 |

**判定标准**：
- 所有必填字段存在且类型正确
- initialBoard 含 32 格（若存在）
- 两个将帅 `visible=true`, `piece="king"`
- 暗子格 `visible=false`

---

### 场景 8：合法 move → moveResult valid=true

**覆盖点**：走子主干——明子移动

**前置条件**：场景 6 完成，当前行棋方=红方（A），局面为开局。

**操作序列**：
1. A 发送：`{"messageType":"move","fromX":"b","fromY":2,"toX":"e","toY":2,"isFlip":true}`
   （炮位暗子 b2→e2 平中，首步 isFlip=true）

**期望消息**：
- A 收到 `moveResult`：`success=true`, `valid=true`, move 信息 + `flipResult`（翻出类型）
- B 收到 `moveResult`：内容同 A（flipResult 双发 Q9），但 capturedPiece 可能不同

**判定标准**：
- valid=true
- 双方均收到 moveResult
- move.fromX/fromY/toX/toY 与实际着法一致
- isFlip 以服务器真值为准

---

### 场景 9：暗子翻子 → flipResult 双发真值

**覆盖点**：翻子机制 Q9——双方均知晓翻出类型

**前置条件**：场景 8 条件（首步暗子走子）。

**操作序列**：与场景 8 相同。

**额外校验**：双方收到的 moveResult 中：
- `flipResult` 字段均存在，值相同，且在合法枚举内（rook/knight/cannon/pawn/guard/bishop）
- 不与棋盘已有明子类型冲突（该方剩余池尚含该类型）

**判定标准**：flipResult 双发一致、类型合法。

---

### 场景 10：吃暗子 → capturedPiece 差异化广播

**覆盖点**：信息保密 Q1/Q13——吃方见真值、被吃方见 "NULL"

**前置条件**：局面设置了吃暗子条件（需在测试中构造特定局面或等待对局中自然发生）。

**操作序列**：
1. 行棋方走出吃对方暗子的着法

**期望消息**：
- 行棋方（吃方）收到 moveResult：`capturedPiece` = 真实类型（如 "rook"）
- 被吃方收到 moveResult：`capturedPiece` = "NULL"

**判定标准**：
- 吃方 capturedPiece ≠ "NULL" 且为合法类型
- 被吃方 capturedPiece = "NULL"

> 注：若被吃子为明子，则双方 capturedPiece 均为真实类型。

---

### 场景 11：非法 move → valid=false + error 2001

**覆盖点**：走法校验——着法违反规则

**前置条件**：局面已开局，轮到红方走。

**操作序列**：
1. 红方发送非法着法（如相/象走田但象眼有子、炮无架直线吃等）

**期望消息**：
- **仅发送方**收到：
  ```json
  {"messageType":"moveResult","success":true,"valid":false,"move":{...}}
  ```
  和
  ```json
  {"messageType":"error","code":2001,"message":"illegal move"}
  ```

- **对方不收到任何消息**（局面不变，仍轮原方）

**判定标准**：
- valid=false，error code=2001
- 仅走子方收到反馈，对方无消息
- 计时不重置

---

### 场景 12：非当前方 move → valid=false + error 2002

**覆盖点**：轮次校验

**前置条件**：局面已开局，轮到红方走。

**操作序列**：
1. 黑方（非行棋方）发送任意 move 消息

**期望消息**：
- 黑方收到 `moveResult` valid=false + `error` code=2002

**判定标准**：
- error code=2002
- 局面不变，仍轮红方

---

### 场景 13：原地翻子拒绝 → valid=false + error 2001

**覆盖点**：禁止原地翻子 Q6/Q11

**前置条件**：局面已开局，某方有暗子。

**操作序列**：
1. 行棋方发送 `from == to` 的 move（如 `fromX="a",fromY=3,toX="a",toY=3,isFlip=true`）

**期望消息**：
- 发送方收到 valid=false + error code=2001

**判定标准**：from==to 一律拒绝（2001）。

---

### 场景 14：将帅被吃 → gameOver checkmate

**覆盖点**：终局——吃到王 Q2

**前置条件**：构造局面使一方可将/帅暴露在对方攻击下。

**操作序列**：
1. 行棋方走出吃对方将/帅的着法

**期望消息**：
- 双方收到：
  ```json
  {"messageType":"gameOver","winner":"red","reason":"checkmate","winnerId":"u1"}
  ```
  （此处 winner 为吃王方颜色，winnerId 为吃王方 userId）

**判定标准**：
- 双方均收到 gameOver
- reason="checkmate"
- 此后 move 消息被忽略

---

### 场景 15：Resign → gameOver

**覆盖点**：终局——认输

**前置条件**：局面已开局。

**操作序列**：
1. 任一方发送 `{"messageType":"Resign"}`

**期望消息**：
- 双方收到：
  ```json
  {"messageType":"gameOver","winner":"black","reason":"resign","winnerId":"u2"}
  ```
  （winner 为**对手**的颜色与 userId）

**判定标准**：
- 双方均收到 gameOver
- reason="resign"
- winner 为认输方的对手

---

### 场景 16：timeout → timeout 消息

**覆盖点**：超时判负 Q14/Q31

**前置条件**：局面已开局，配置 `turnTimeoutMs` 可临时设为短值（如 5000ms）以加速测试。

**操作序列**：
1. 行棋方在超时阈值内不发送任何 move

**期望消息**：
- 双方收到：
  ```json
  {"messageType":"timeout","loserId":"u1","winnerId":"u2","reason":"timeout"}
  ```

**判定标准**：
- 双方均收到 timeout
- loserId = 当前行棋方 userId
- 房间进入 FINISHED 状态
- 迟到 move 被拒绝（error 2003）

---

### 场景 17：disconnect → gameOver

**覆盖点**：断线判负

**前置条件**：局面已开局。

**操作序列**：
1. 一方主动关闭 WebSocket 连接

**期望消息**：
- 对方收到：
  ```json
  {"messageType":"gameOver","winner":"black","reason":"disconnect","winnerId":"u2"}
  ```
  （winner 为在线方的颜色和 userId）

**判定标准**：
- 在线方收到 gameOver reason="disconnect"
- 房间进入 FINISHED 状态

---

### 场景 18：终局后消息被忽略

**覆盖点**：状态守卫——FINISHED 房间不接受新 move

**前置条件**：gameOver 已发出（任一种终局）。

**操作序列**：
1. 任一方发送 `move` 消息

**期望消息**：无响应（move 被静默忽略并记日志）

**判定标准**：无 moveResult 回复，无异常。

---

### 场景 19：非法 JSON / 缺 messageType → error 4001

**覆盖点**：协议容错——解析失败

**前置条件**：客户端已连接（无需登录）。

**操作序列**：
1. 发送 `{invalid json`
2. 发送 `{"noType":"here"}`（缺失 messageType）

**期望消息**：
```json
{"messageType":"error","code":4001,"message":"JSON format error"}
```

**判定标准**：收到 error 4001，连接不关闭。

---

### 场景 20：未知 messageType → 静默忽略

**覆盖点**：宽容解析 §4.9-2——兼容未来扩展

**前置条件**：客户端已连接。

**操作序列**：
1. 发送 `{"messageType":"unknownFutureMsg","data":"hello"}`

**期望消息**：无任何响应（服务器静默忽略 + 写日志）

**判定标准**：连接保持，不崩溃，不回 error。

---

### 场景 21：入站帧 >1KB → 断连

**覆盖点**：Q27 守卫——超限报文丢弃并关闭连接

**前置条件**：客户端已连接。

**操作序列**：
1. 发送一个超过 1024 字节的 UTF-8 文本帧

**期望消息**：连接被服务器关闭

**判定标准**：WebSocket 连接关闭（onClose 回调触发）。若在 PLAYING 中 → 按断线终局。

## 错误码速查（§4.6 / 公共接口）

| code | 含义 | 触发场景 |
|---|---|---|
| 1001 | 登录失败（账号或密码错误） | Login 时凭据无效 |
| 1002 | 重复登录 | 同 userId 已在线 |
| 2001 | 非法走子（规则不符） | 场景 11、13 |
| 2002 | 未轮到本方走子 | 场景 12 |
| 2003 | 超时未走子 | 场景 16（迟到 move） |
| 3001 | 房间不存在 | 内部异常兜底 |
| 3002 | 匹配失败 | cancelMatch 后 |
| 4001 | JSON 格式错误 | 场景 19 |

## 消息大小写约定（§4.9 宽进严出）

| 方向 | 规则 |
|---|---|
| C→S 发送 | 严格照公共接口字面：`Login`/`Ready`/`Resign` 首字母大写，其余全小写 |
| S→C 发送 | 字段名、大小写、类型严格照公共接口字面 |
| 解析入站 | messageType 匹配忽略大小写；未知类型静默忽略 |
